# 50 VT (Virtual Terminal)

DRAFT 2026-05-09. Dep:`01`,`02`,`07`,`13`,`15`,`16`,`19`,`28`,`46`,`47`,`49`. Provides:`/dev/tty0..6`,`/dev/console`,`/dev/tty` (controlling),VT switching `/sys/class/tty/tty0/active`.

## 1 Purpose

Linux Virtual Terminal layer per `linux/drivers/tty/vt/vt.c`. Multiplexes 6 text consoles (`/dev/tty1..tty6`) sharing a single graphical (`49` fbcon) or serial (`28` tty) backend. Provides `KDSETMODE` / `KDGETMODE` (text vs graphics), `KDGKBMODE` / `KDSKBMODE` (raw vs translated keyboard), `VT_OPENQRY` / `VT_GETSTATE` / `VT_ACTIVATE` / `VT_WAITACTIVE` / `VT_RELDISP` for orderly VT switching used by getty/login/Xorg/Wayland.

## 2 Invariants (frozen)

1. `/dev/tty1..tty6` ino base = `0x70020001..0x70020006`. devfs registers all 6 at boot.
2. `/dev/tty0` is an alias for the foreground VT — writes go to whichever VT is active. ino = `0x70020000`.
3. `/dev/tty` resolves per-process to the controlling terminal (Linux `current->signal->tty`); ino base = `0x70020100` + tid lookup.
4. `/dev/console` ino = `0x70020080`, also writes to the foreground VT but bypasses session/pgrp checks (kernel klog uses this).
5. Exactly one VT is "foreground" at a time; switching is `VT_ACTIVATE` from a master process (kernel `chvt(N)` syscall equivalent runs over the same path).
6. Each VT has independent state: cursor pos, scrollback, attrs, line discipline, foreground/background pgrp, mode (KD_TEXT/KD_GRAPHICS), keyboard mode (K_XLATE/K_RAW/K_MEDIUMRAW/K_UNICODE).
7. When a VT is in `KD_GRAPHICS`, the kernel STOPS rendering glyphs to its FB; userspace owns the screen until `KDSETMODE(KD_TEXT)` or VT switch. Used by Xorg/Wayland for direct DRM rendering.
8. Hangup (`SIGHUP` to controlling pgrp) fires on tty disconnect AND on switch-away IF the VT was running a session (per `28§4`).

## 5 vt_mode struct

```c
// linux/include/uapi/linux/vt.h
struct vt_mode {
    char  mode;       // VT_AUTO=0 (kernel switches) / VT_PROCESS=1 (signal owner)
    char  waitv;      // unused
    short relsig;     // signal to send asking owner to release
    short acqsig;     // signal sent after switch is granted
    short frsig;      // unused
};

struct vt_stat {
    unsigned short v_active, v_signal, v_state;
};
```

VT_PROCESS protocol (used by Xorg during keypress Ctrl+Alt+Fn):
1. Kernel notes switch request to VT M.
2. Kernel sends `relsig` (typically `SIGUSR1`) to VT N's owner.
3. Owner does cleanup (un-mmap fbdev, save state) and calls `VT_RELDISP(1)` to grant, or `VT_RELDISP(0)` to deny.
4. On grant, kernel switches; new VT's owner receives `acqsig` (typically `SIGUSR2`).
5. On deny, switch is aborted; user sees a beep.

## 6 Keyboard input flow

```
46 virtio-input  ──events──>  vt::deliver_input_event
                                      │
                            ┌─────────┴──────────┐
                            │ K_XLATE (default)  │
                            │ keymap → keysym    │
                            │ keysym → utf8      │
                            └─────────┬──────────┘
                                      │
                                      v
                          active-VT line discipline (28)
                                      │
                                      v
                          read(/dev/tty<N>) returns bytes
```

K_RAW: forward `KEY_*` codes as 1-byte scancodes to the reader (preserves `0x80` make/break bit).
K_MEDIUMRAW: 2/3-byte sequences, identifies high-numbered keys via continuation bit; used by older console subsystems.
K_UNICODE: same as K_XLATE but UTF-8 multi-byte for non-ASCII; default on Linux distros today.
K_OFF: discard input entirely; used when X owns the kbd via evdev directly.

## 7 Keymap (en_US-default v1)

Compiled-in `keymap.rs` is a const `[KeymapEntry; 256]` where each entry has:
```
plain: u32,   // keysym for unmodified press
shift: u32,
ctrl:  u32,
alt:   u32,
shift_alt: u32,
ctrl_alt:  u32,
```
Keysym values follow `linux/include/uapi/linux/keyboard.h` (KEY_A=`(K_LATIN<<8)|'a'` etc.). Only en_US ships v1; locale-specific keymaps (de, fr, dvorak, ...) ride v2.x via `KDSKBENT` to upload at runtime.

Special keys handled directly by the VT layer (NOT forwarded to readers):
- `Ctrl+Alt+F<N>` → `VT_ACTIVATE(N)`
- `Ctrl+Alt+Del` → `kernel_panic_request()` (Linux: send SIGINT to init; v1: log + halt)
- `Shift+PageUp` / `PageDown` → fbcon scrollback (per `49§8`)
- `Caps Lock` toggles caps-LED via `KDSETLED`
- `Num Lock` similarly
- `Scroll Lock` pauses fbcon output

## 8 VT switching sequence (auto-mode)

1. Active = N; request switch to M (M ≠ N, 1 ≤ M ≤ 6).
2. If VT M is in `KD_GRAPHICS` and has VT_PROCESS mode, run §5 protocol.
3. Else: pause N's fbcon, swap fbcon's per-VT state, redraw M's backing buffer to scanout.
4. Update `/sys/class/tty/tty0/active` to "ttyM".
5. Send keyboard focus to M.

## 9 Concurrency

- Global `Spinlock<VtState>` guards active-id + per-VT slot table.
- Per-VT `Spinlock<VtSlot>` guards each VT's mode/scrollback/etc.
- Lock order: VtState → VtSlot[N] → fbcon → DRM. Never invert.
- Input delivery (`46` callback) acquires VtState read-only to learn active id, then VtSlot[active] for line-discipline write.

## 10 Failure modes

- `VT_ACTIVATE` for out-of-range or DISALLOCATE'd VT: `EINVAL`.
- `VT_RELDISP(0)` (deny) during a switch: switch aborts, audible bell.
- `KDSETMODE(KD_GRAPHICS)` while another VT also has GRAPHICS: still permitted (each VT has its own backing buffer); but only the foreground VT's buffer is bound to scanout.
- read on a VT whose tty is hung-up: `EIO`.

## 11 Failure modes — input layer

- Keymap lookup miss (unmapped scancode): drop key, no read-side bytes.
- `KDSKBMODE(K_RAW)` while no userspace reader exists: events accumulate in the line-disc buffer until full, then drop with stat counter increment.

## 13 Cross-spec
